# 01. 產品需求文件 (PRD)

> **版本**:v1.0  
> **狀態**:已確認(Phase 0 基準版)  
> **最後更新**:專案啟動

---

## 1. 產品願景

打造一套以「**學生即專案**」為核心的 CRM 系統,讓留學代辦公司的所有顧問能在統一平台上協作,從**招生 → 諮詢 → 成交 → 申請 → 入學**全流程數位化管理。

---

## 2. 服務領域

- **主力業務**:美、英、加、澳的大學/碩士留學申請
- **副線業務**:語言學校、遊學團

---

## 3. 使用者角色

### 3.1 角色階層(4 階)

```
顧問(Consultant)──┬── 前端顧問(Frontend) ─ 招生
                   └── 後端顧問(Backend)  ─ 行政
        ↓
前端主管(Manager - Frontend) | 後端主管(Manager - Backend)
        ↓
老闆(Admin)
```

### 3.2 角色職責對照

| 角色 | 主要工作 | 系統權限重點 |
|---|---|---|
| 前端顧問 | 開發名單、諮詢、成交、選校表 | 看自己學生、新增/編輯成交、製作選校表 |
| 後端顧問 | 文件、字數、Portal、申請追蹤 | 看自己學生、文件版控、申請進度 |
| 前端主管 | 督導招生、看績效 | 看所有學生、覆核成交、看招生數據 |
| 後端主管 | 督導行政、Workload 分派 | 看所有學生、調派後端顧問、看執行品質 |
| 老闆 | 全公司營運 | 全權限、營收、系統設定、人員管理 |

## 4. 核心功能模組

### 4.1 招生模組(前端顧問主用)

- 新增/編輯學生名單
- 學生分級與狀態管理
- 名單來源追蹤(自開發、行銷部、轉介)
- 成交流程:選方案、加購字數/學校、優惠、績效拆分
- 選校表製作(多版本、滾動式調整)
- 顧問交接給後端

### 4.2 行政模組(後端顧問主用)

- 各申請學校 Portal 帳密管理(加密)
- 文件版本控制(Master + 學校客製版本 + 多版本歷史)
- 字數帳本(初始額度 / 加購 / 養字 / 扣減,可追溯)
- 學業成績追蹤(GPA、TOEFL/IELTS、GRE/GMAT)
- 各校申請進度追蹤(待送件 / 已送件 / 補件 / 結果)
- 合作學校回傭紀錄

### 4.3 共用模組

- 學生 360° 視圖(集中所有資訊與時間軸)
- 儀表板(個人 / 團隊 / 全公司)
- 公告系統
- 工作量能(Workload)計算
- 通知系統

### 4.4 後續擴充(Phase 6 以後)

- 院校資料庫(從現有 Beta 整合)
- 歷屆榜單(自動從已入學學生統計)
- 知識庫
- 問題討論與回報
- 營收儀表板
- AI 整合:落點分析、智能推薦學校(透過 Claude API + MCP)

## 5. 業務流程關鍵點

### 5.1 學生狀態流(13 狀態)

```
[招生階段] 新名單 → 聯繫中 → 諮詢中 → 意向客戶 → 已成交
                                            ↓
[申請階段] 資料準備 → 選校規劃 ⇄ 書審準備 → 申請送出
                                            ↓
                            等待結果 → 錄取確認 → 入學準備 → 已入學

[特殊狀態] 暫緩(可恢復) | 退費終止(終止失敗) | 無效名單(招生階段終止)
```

### 5.2 成交與績效拆分

- **預設情境**:單一顧問成交 → 100% 給該顧問
- **轉介情境**:來自網路行銷部、其他顧問介紹、外部品牌 → 顧問 65% / 轉介人 35%
- 拆分比例可手動調整

### 5.3 字數帳本

```
初始額度(來自方案)
  + 加購字數(成交時加購)
  + 養字(顧問加碼)
  - 修改扣減(每次文件 diff)
  ─────────────────────────
  = 當前可用餘額
```

每筆交易都記錄,可完整追溯。

### 5.4 文件版本流

```
Master 主版本(學生層級)
   │  例:Master SOP v1, v2, v3...
   │
   ├─ Fork(拷貝出客製版本,不扣字數)
   │
   ▼
Variant 客製版本(學校層級)
   例:哥大 EE SOP v1, v2, v3... (扣字數)
        史丹佛 EE SOP v1, v2... (扣字數)
```

## 6. 非功能需求

### 6.1 安全性

- Portal 密碼必須加密儲存(AES-256)
- 所有寫入操作必須通過 RLS 驗證
- 學生個資需符合台灣個資法基本要求(可匿名化匯出、可刪除)
- 操作軌跡可追溯(activity_log)

### 6.2 可用性

- 9-22 點工作時段保證可用
- 計畫月停機 < 2 小時(維護時段)

### 6.3 規模假設

- 顧問:15-50 人並行
- 學生主檔:預估 5 年內 < 10,000 筆
- 文件版本:預估每位學生 50-200 個版本
- 同時上線:< 30 人

### 6.4 效能

- 學生列表載入 < 1 秒
- 學生 360° 視圖載入 < 2 秒
- 文件比對 / 字數計算 < 500ms

## 7. 範圍外(明確排除)

- ❌ 學生端介面(學生不登入此系統)
- ❌ 公開招生網站(這是內部 CRM)
- ❌ 行銷部投廣告系統(他們用自己的工具)
- ❌ 會計/發票系統(僅記錄金額,不開發票)
- ❌ HR 系統(僅權限管理,不管考勤)
